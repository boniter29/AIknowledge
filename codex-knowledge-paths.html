Codex 访问知识与文件的路径对比

1. Codex 读取钉钉、GitHub、本地 workspace 的路径

数据源 Codex 典型访问路径 本质
本地 workspace Agent → Shell / 文件工具 → rg / ls / 读取文件 → 选择性放入上下文 工具访问 + 精确文件读取
GitHub Agent → GitHub Connector / gh CLI / 本地 git 仓库 → PR、issue、diff、文件、CI 日志 远程 API 访问 + 结构化检索
钉钉 Agent → dws / 钉钉能力工具 → 文档、群聊、日程、审批、知识库等接口 业务系统 API 访问 + 权限内查询
已安装 Knowledge Skill 的知识 Agent → Knowledge Skill → 切片 → 向量化 → 召回 → 再读取/引用 RAG 语义检索

2. 这些路径的效率区别

路径 优势 代价 适合场景
工具访问 + 文件读取 准确、实时、可验证，适合代码和具体文件 需要 Agent 多轮搜索和阅读，语义模糊查询效率一般 代码仓库、配置文件、日志、PR diff、本地项目
Connector / API 访问 权限清晰，能访问动态业务数据 依赖接口能力、速率限制、搜索质量 钉钉、GitHub、日历、审批、群聊、远程文档
RAG 适合海量自然语言知识的语义召回 需要索引、切片、向量化、刷新；召回是概率性的 制度文档、FAQ、历史知识、跨文档问答
直接塞进上下文 模型可直接综合全部内容 token 成本高，长上下文会稀释注意力 小规模材料、一次性分析

3. 当前 harness 架构中是否有必要做 Knowledge Skill

如果目标只是让 Codex 读取 workspace 文件，没有必要专门做 Knowledge Skill。现有 Codex 路径已经可以通过文件工具、shell、rg、GitHub Connector、钉钉工具等方式读取需要的内容。

但如果目标是把大量稳定知识沉淀成“可语义搜索的组织知识库”，Knowledge Skill 就有价值。它不是替代文件读取，而是补充一层语义召回能力。

建议做 Knowledge Skill 的情况

  • 资料很多，用户经常问“某个制度怎么规定”“以前有没有类似案例”。
  • 知识以自然语言为主，而不是代码结构为主。
  • 需要跨 workspace、钉钉文档、GitHub 文档、历史资料统一检索。
  • 用户不知道关键词，只能用语义描述问题。
  • 需要跨会话保留一套长期知识索引。

不建议优先做的情况

  • 主要处理代码仓库。代码更适合 rg、AST、符号搜索、测试和 git diff。
  • 文件变化很频繁，索引刷新成本高。
  • 问题通常能通过文件名、函数名、错误日志、PR 编号精确定位。

4. Knowledge Skill 和现有 Codex 文件访问的区别

能力 现有 Codex 文件访问 Knowledge Skill
发现方式 Agent 主动搜索、读目录、读文件、看 diff 先语义召回相关切片
索引 默认不把整个 workspace 向量化 会建立切片和向量索引
准确性 读取原文，适合精确验证 召回相关片段，可能漏召或误召
适合内容 代码、配置、日志、结构化文件 文档、FAQ、制度、知识沉淀
更新成本 实时读取最新文件 需要重新索引或增量更新

5. Codex 怎么在海量上下文里找出需要的内容

Codex 通常不是把所有内容一次性塞进模型上下文，而是采用“工具驱动的渐进式工作集”：

  1. 先理解任务，判断可能涉及哪些数据源。
  2. 用 rg、文件列表、GitHub 搜索、钉钉查询等工具缩小范围。
  3. 读取少量高相关文件或片段进入上下文。
  4. 根据线索继续扩展到相邻文件、调用链、PR、日志或文档。
  5. 最后用原始来源验证结论，而不是只相信召回摘要。

结论

当前 harness 里，Codex 的默认路径更像“Agent + Tool Calling + Source Access”，不是默认 RAG。Knowledge Skill 的价值在于给海量自然语言知识增加语义检索层。

推荐采用混合架构：代码、本地 workspace、GitHub PR、日志继续走工具访问；稳定文档、制度、FAQ、历史知识、钉钉知识沉淀可以做 Knowledge Skill。这样既保留 Codex 的精确读取能力，也补上大规模知识问答的召回效率。
